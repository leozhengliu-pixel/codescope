# Acceptance Spec: Generic/local Git parity

## Purpose
This document creates the dedicated black-box acceptance home for generic Git host and local-path repository parity. It freezes the current rewrite contract around authenticated connection management, generic/local connection metadata, and the user/operator-visible gaps that still remain before self-hosted local mirrors count as first-class Sourcebot parity.

## Grounding and limits
- This is an acceptance contract for the currently evidenced generic/local Git surface, not a claim that end-to-end local mirror parity is already shipped.
- Grounding for this document is limited to live rewrite evidence in:
  - `crates/models/src/lib.rs`
  - `crates/api/src/main.rs`
  - `crates/api/src/storage.rs`
  - `crates/core/src/lib.rs`
  - `web/src/App.tsx`
  - `web/src/App.test.tsx`
  - `specs/acceptance/index.md`
  - `specs/acceptance/integrations.md`
  - `specs/FEATURE_PARITY.md`
  - `docs/reports/2026-04-18-parity-gap-report.md`
  - `docs/plans/2026-04-18-sourcebot-full-parity-roadmap.md`
- When the rewrite exposes connection records, settings-shell CRUD, one minimal settings-driven local import form for existing `local` connections, a single authenticated local-path import baseline that can hand off one explicitly requested local Git working tree into the PostgreSQL catalog, bind it to the importing admin organization, and queue one repository-sync job for the worker baseline, a bounded admin-only repository-sync enqueue API for already visible repositories, a fail-closed retry API that requeues failed visible jobs without overwriting terminal history, active-job suppression that rejects duplicate queued/running jobs for the same organization/repository/connection target, a one-shot worker smoke that advances an enqueued job into authenticated terminal history, bounded stale-running repository-sync lease recovery plus one automatic stale-lease retry before the next claim, and a timeout-bounded worker-side local-Git preflight plus bounded tracked-content discovery for `local` connection jobs, this spec records that truthfully while still deferring broader local mirror enumeration, real fetch/import/reindex execution, broad automated retries/backoff, recursive/provider import, and full durable catalog parity to later slices.

## Scope
- Authenticated connection CRUD for generic Git host and local-path records
- One authenticated local-path repository import baseline for a configured `local` connection
- User-visible generic/local connection metadata in repository detail and settings surfaces
- Acceptance boundaries for self-hosted local mirrors versus the still-missing real ingestion/runtime path
- A bounded admin-only repository-sync enqueue/retry contract for already visible generic/local repositories that rejects duplicate queued/running jobs for the same organization/repository/connection target, plus a repo-local worker smoke proving that a queued job becomes authenticated terminal sync-history progress under the current stub/local-preflight worker baseline

## Current rewrite evidence anchors

- `crates/api/src/main.rs` now accepts authenticated admin-only `POST /api/v1/auth/repository-sync-jobs` for an already visible `(organization_id, repository_id)`, derives the job's `connection_id` from catalog repository detail, persists a queued repository-sync job, exposes terminal `synced_revision`, `synced_branch`, and `synced_content_file_count` metadata on authenticated sync-history responses, and fails closed for non-admin callers without mutating sync-job state. ## Inputs
- Authenticated local-session context for connection management and sync-history visibility
- Connection kind (`generic_git` or `local`)
- Generic host configuration (`base_url`) or local mirror configuration (`repo_path`)
- Authenticated local import payloads with `connection_id` plus a requested repository path under the configured local root
- Repository detail lookup for repo-scoped connection metadata
- Read-only repository sync-job history for the configured connection
- Admin-only repository sync-job enqueue payloads with `organization_id` plus `repository_id` for repositories the caller can already administer and see

## Expected behavior
1. An authenticated admin/settings user can list existing connection records, including generic Git host and local-path variants.
2. Creating or updating a `local` connection uses a `repo_path` configuration contract rather than a SaaS-style `base_url`.
3. Creating or updating a `generic_git` connection uses a `base_url` configuration contract rather than a local path.
4. An authenticated admin/settings user can import one real local Git working tree whose canonical path stays under a configured `local.repo_path`, and the API returns repository detail wired to that local connection.
5. After that import, the repository appears through the normal repository list/detail surfaces with the associated connection metadata, including connection name and kind.
6. The settings connections shell can show read-only repository sync history alongside the connection inventory without implying that full local/generic ingestion parity is already complete.
7. An authenticated organization admin can enqueue a queued repository-sync job for an already visible repository and can requeue a failed visible job without overwriting its terminal history, but both paths fail closed with `409 Conflict` without appending a job when a queued/running sync already exists for the same organization/repository/connection target; the repo-local smoke matrix proves that the current worker can advance a queued job into authenticated `succeeded` sync-history progress, the worker marks stale `running` repository-sync jobs failed after a one-hour lease before claiming the next queued job, can queue one automatic stale-lease retry after an additional one-hour backoff when no active or prior automatic replacement exists, an explicitly configured bounded multi-tick worker invocation can process multiple queued jobs before exiting, and jobs tied to configured `local` connections now perform a timeout-bounded real Git working-tree preflight plus bounded revision/content-discovery/current-branch probes that fail closed for missing, invalid, empty, empty-tree, or hung paths and succeed for real local Git repositories with a readable HEAD, at least one tracked content path, and a current branch or detached `HEAD` label while persisting terminal revision/branch/content-count metadata, writing a repo-local tracked-content manifest, materializing a sibling snapshot tree from tracked `HEAD` file bytes, and persisting a sibling `search-index.json` artifact that can be loaded without rewalking the snapshot; explicitly scoped and unscoped/search-context search plus index-status requests can load the latest successful local sync `search-index.json` artifact (falling back to the snapshot tree only for pre-artifact rows) while no-revision tree/blob reads for that visible repository can use the latest successful local sync snapshot and commit list/detail/diff reads can use the latest successful local Git working tree from an authorized `(organization_id, repository_id)` boundary when the startup search/browse/commit stores have no indexed rows, browseable tree/blob, or seeded commit history for it, but this remains a bounded local artifact-backed search/index-status plus snapshot browse/code-navigation and commit-history baseline rather than real provider fetch/import/reindex execution, broad automated retry/backoff, or production scheduling/supervision parity.
8. The rewrite must not claim generic/local Git parity solely because the models, settings shell, one minimal settings-driven local import UX, one local import baseline with PostgreSQL catalog handoff, admin-organization visibility binding, and queued repository-sync handoff for an explicitly requested local working tree, read-only sync history, bounded manual sync enqueue/retry with active duplicate suppression, bounded worker progress smoke/multi-tick loop, local Git preflight plus persisted revision/content-count/current-branch status handling, bounded tracked-content manifest/snapshot/search-index artifact, Generic Git remote `ls-remote --symref HEAD` default-branch revision/current-branch metadata handling only when the selected HEAD branch/revision is also advertised by `--heads`, direct-HEAD-revision branch matching, ambiguous duplicate-HEAD fail-closed parsing, and `ls-remote --heads` fallback, stale-running lease recovery plus one automatic stale-lease retry, and requested-repo artifact-backed search/index-status, unscoped/search-context search artifact fallback, and snapshot tree/blob/code-navigation fallback exist; broader host enumeration, recursive import, clone/fetch/import/reindex execution, durable/background indexing, provider discovery, provider supervision, and full durable catalog-backed provider visibility still have to be proven.

## Permission behavior
- Generic/local connection CRUD requires an authenticated local session and follows the same auth/admin settings boundary as the existing `/api/v1/auth/connections` surface.
- Repository detail and sync-history visibility remain constrained by the caller's current repository visibility; generic/local metadata must not bypass the normal repo-visibility model.
- The local import baseline must fail closed for non-admin users, for non-`local` connections, and for requested paths outside the configured local connection root.
- Repository-sync enqueue and retry must fail closed for non-admin users, hidden repositories, blank payload fields, catalog-missing repositories, non-failed retry targets, or an existing queued/running job for the same organization/repository/connection target, and failure cases must not append queued jobs.
- Hidden repositories or cross-org invisible sync jobs must fail closed rather than leaking through generic/local connection surfaces.

## Edge cases
- Local-path connection UX is distinct from SaaS/git-host providers: `repo_path` is the visible editable contract, not `base_url`.
- Generic/local connection records can exist before the rewrite proves real repository enumeration or indexing for those hosts.
- The first local import baseline is intentionally narrower than full provider parity: it supports one explicitly requested local working tree, not recursive repository enumeration or generic-host discovery.
- Read-only sync history can be visible for a connection even while the durable catalog-backed repository list/detail parity remains incomplete.
- Manual sync enqueue can create a queued job for an already visible repository, manual retry can append a new queued attempt for a failed visible job while preserving terminal history, both paths reject duplicate queued/running jobs for the same organization/repository/connection target without mutation, the default worker smoke can advance that stub job to terminal authenticated history, an explicit `SOURCEBOT_WORKER_MAX_TICKS` bounded worker invocation can process multiple queued jobs before exiting, stale running repository-sync jobs are marked failed after a one-hour lease before the next claim, one automatic stale-lease retry can be queued after an additional one-hour backoff when no active or prior automatic replacement exists for the same target, local-connection jobs now preflight the configured path as a timeout-bounded real Git working tree and probe readable HEAD, tracked content paths, and current-branch metadata before persisting terminal revision/branch/content-count fields and local-only artifacts, and Generic Git jobs now reject blank, option-like, control-character-bearing, or credential-bearing HTTP(S) `base_url` before spawning Git, probe `git ls-remote --symref -- <base_url> HEAD`, use an advertised symref default branch when present, otherwise map a direct `HEAD` revision to its matching advertised branch before falling back to the first advertised `git ls-remote --heads -- <base_url>` branch only when all advertised head lines are well-formed, and persist terminal revision/current-branch metadata while failing closed for malformed or oversized advertisements, including unexpected non-`HEAD` records in the HEAD metadata probe, and terminating the child promptly once retained stdout/stderr exceeds the configured cap; none of this is evidence that a provider worker can discover repositories, clone/fetch/import/reindex contents, perform broad automated retry/backoff, or maintain durable index status.
- Fixture-backed or seeded repository detail metadata is not sufficient evidence that adding a new local repository path triggers a real import/index cycle; the import baseline must prove a real Git working tree can be surfaced through the catalog.

## Black-box examples
- Opening `#/settings/connections` shows existing `generic_git` and `local` connection records in the authenticated settings inventory.
- For an existing `local` connection, `#/settings/connections` exposes a minimal per-connection import form for one explicit repository path, posts the selected `connection_id` plus path to `/api/v1/auth/repositories/import/local` (API callers can include `organization_id` such as `org_acme` when more than one admin organization is available), renders the imported repository name/id on that same card after success, refetches authenticated repository-sync history so the queued worker handoff appears on the same local connection card, and shows scoped import failures on that same card without adding generic-host discovery or broader sync controls.
- Creating a local mirror connection sends a nested request with `kind: "local"` and `config: { "provider": "local", "repo_path": "/srv/git/mirror" }` rather than a host-style `base_url` payload.
- Editing a generic Git host connection preserves the host-style `base_url` contract.
- Opening `#/repos/:repoId` shows repository detail with connection name/kind metadata for an already known repository.
- Posting `{"organization_id":"org_acme","connection_id":"conn_local_acme","path":"/srv/git/local/project"}` to `/api/v1/auth/repositories/import/local` for an authenticated admin imports that real local Git working tree, then `/api/v1/repos` and `/api/v1/repos/:repoId` surface it with the associated local connection metadata.
- An authenticated user with repository visibility can inspect read-only sync history for repositories tied to a configured connection, but that does not by itself prove the rewrite can newly enumerate or ingest repositories from that host.
- An authenticated organization admin can post `{"organization_id":"org_acme","repository_id":"repo_visible"}` to `/api/v1/auth/repository-sync-jobs` and receive a `queued` repository-sync job for the repository's catalog connection; after review-agent work is drained, the default worker can advance that job to `succeeded` authenticated sync history with `started_at` and `finished_at`, and `SOURCEBOT_WORKER_MAX_TICKS=2 SOURCEBOT_WORKER_IDLE_SLEEP_MS=0` can process two queued jobs in one explicitly bounded invocation before logging bounded runtime completion; if a visible failed job exists, the admin can post to `/api/v1/auth/repository-sync-jobs/{job_id}/retry` and receive a fresh queued retry while the failed row remains intact; for jobs tied to a configured `local` connection, a missing or hung preflight records a `local repository sync preflight failed` terminal failure, an empty, empty-tree, or hung local repository records a `local repository sync execution failed` terminal failure, and a real local Git repository with readable HEAD, at least one tracked content path with no ASCII control characters, and either current-branch metadata or detached `HEAD` records terminal success plus synced revision/branch/content-count metadata visible through authenticated history; the settings sync-history row computes the matching `.sourcebot/local-sync/<organization_id>/<repository_id>/<job_id>/manifest.txt` and sibling `snapshot/` artifact locations for local operators, and no-revision `/api/v1/repos/{repo_id}/tree`/`blob` can browse/read files, no-revision `/api/v1/repos/{repo_id}/definitions`/`references` can resolve bounded symbol locations from that latest authorized local snapshot, and `/api/v1/repos/{repo_id}/commits` plus commit detail/diff routes can read commit history from the latest successful authorized local Git working tree when the startup browse/code-navigation/commit stores have no tree/blob, primary blob, or seeded commit history for the visible repository, and `/api/v1/repos/{repo_id}/index-status` can report bounded indexed-file/line counts from the same local sync `search-index.json` artifact when present (falling back to the snapshot only for pre-artifact rows) when the startup search store has no status for the visible repository before any broader fetch/import/reindex work exists.

## Deferred parity gaps locked by this spec
This spec intentionally records the remaining gaps instead of over-claiming parity:

1. **Generic-host and recursive local enumeration parity are still missing.** The current rewrite evidence now covers one authenticated local-path import baseline, but not generic-host discovery or recursive import of many repositories from a connection root.
2. **Durable catalog parity is still partial.** Repository list/detail reads and one explicit local Git import handoff can use the PostgreSQL catalog path, but recursive local enumeration, generic-host/provider discovery, connection-management persistence, real fetch/import/reindex execution, and full durable visibility/reconciliation semantics remain blocked.
3. **Per-connection sync/index parity is still shallow.** The settings shell now includes a minimal local-only explicit-path import form, read-only sync history, a backend-only admin enqueue/retry baseline for already visible repositories, a default one-tick worker progress smoke plus explicit bounded multi-tick worker coverage, timeout-bounded local-Git preflight plus persisted revision/content-count/current-branch status handling with bounded local-only tracked-content manifest/snapshot/search-index artifacts with fail-closed rejection of ASCII-control tracked paths before artifact persistence, requested-repo artifact-backed search/index-status, unscoped/search-context search artifact fallback, snapshot tree/blob/code-navigation fallback, local-Git commit list/detail/diff fallback, and a Generic Git metadata-only `ls-remote --symref HEAD` default-branch worker probe with `ls-remote --heads` fallback, but not full operator controls, clone/fetch/import/reindex execution, automated retries/backoff, import progress, generic-host discovery, recursive local enumeration, durable/background indexing, or durable per-connection indexing status parity.
4. **Broader provider/runtime parity is still deferred.** GitLab, GitHub, and later multi-host provider work still need real auth/configuration, enumeration, sync execution, webhook/runtime behavior, and provider-specific acceptance follow-ups.
5. **Settings/admin navigation parity is still incomplete.** The rewrite has a limited `#/settings/connections` shell, but not the broader auth/admin/settings route family that would make generic/local connection management first-class product navigation.
