# Worker Runtime Acceptance

## Purpose
This document defines the current clean-room acceptance contract for the `sourcebot-worker` runtime in `crates/worker/src/main.rs`. It describes the shipped one-tick default plus bounded multi-tick, mostly stub-oriented worker baseline with bounded local-Git and Generic Git metadata probes, without over-claiming real fetch/import/reindex execution, retries, production scheduling, durable worker metadata, or production observability.

## Grounding and limits
- Grounded in:
  - `crates/worker/src/main.rs`
  - `crates/worker/src/lib.rs`
  - `crates/worker/tests/worker_runtime_logging_smoke.rs`
  - existing focused worker smoke coverage in `crates/worker/tests/*.rs`
- By default, the worker remains a one-tick local bring-up path. Setting `SOURCEBOT_WORKER_MAX_TICKS=<n>` lets one invocation run a bounded multi-tick loop and then exit; oversized values above 1,000,000 fail closed before execution. `SOURCEBOT_WORKER_IDLE_SLEEP_MS=<ms>` controls the sleep between no-work ticks and fails closed above 3,600,000 milliseconds (one hour) so operator typos cannot park the bounded loop for days.
- The repo-local smoke matrix now proves the API enqueue/history surface can hand a queued repository-sync job to a later `sourcebot-worker` invocation; authenticated local import also queues a newly imported local repository for that same worker baseline after adding the imported repository to the admin organization visibility set. The terminal `succeeded` progress remains observable through the authenticated history API and persisted organization state.
- Repository-sync jobs tied to a configured `local` connection now validate that `repo_path` is non-blank, absolute, and does not contain control characters or surrounding whitespace before spawning Git, then run a timeout-bounded real Git preflight (`git -C <repo_path> rev-parse --is-inside-work-tree`) plus bounded revision, content-discovery, current-branch, detached-branch-label, and tracked-file snapshot probes (`git rev-parse HEAD`, `git ls-tree -rz --name-only HEAD`, `git symbolic-ref --short HEAD`, fallback `git rev-parse --abbrev-ref HEAD`, and `git show HEAD:<tracked-path>`) whose retained stdout/stderr are capped by the same combined-output guard before falling back to stub outcomes: blank, padded, control-character-bearing, missing, invalid, empty, empty-tree, detached, malformed-HEAD-revision, hung, or combined stdout/stderr-over-cap preflight local repositories fail closed with operator-visible `local repository sync preflight failed` or `local repository sync execution failed` errors instead of claiming success. Successful local jobs write a deterministic repo-local tracked-content manifest plus bounded text snapshot/search-index artifacts under `.sourcebot/local-sync/...` for tracked non-binary files only; tracked paths with tabs are preserved as valid Git path names while non-UTF-8 path bytes and manifest-delimiting/control characters such as newlines still fail closed before artifact success.
- Repository-sync jobs tied to a configured `generic_git` connection now validate that `base_url` is non-blank, uses only the bounded `http`, `https`, `ssh`, `git`, `file`, or scp-like remote forms, rejects unsupported transport prefixes such as Git's `ext::` before spawning Git, and verifies that HTTP(S), SSH-URL, and scp-like remotes do not include embedded credentials/userinfo before spawning Git, then run timeout-bounded real `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` metadata-only probes with bounded stdout/stderr handling and non-interactive Git auth environment overrides (`GIT_TERMINAL_PROMPT=0`, `GCM_INTERACTIVE=never`, `GIT_ASKPASS=/bin/false`, `SSH_ASKPASS=/bin/false`, `SSH_ASKPASS_REQUIRE=never`, and `GIT_SSH_COMMAND=ssh -oBatchMode=yes -oNumberOfPasswordPrompts=0`) before falling back to stub outcomes. Reachable remotes with at least one advertised branch persist terminal revision/current-branch metadata from the default branch when the HEAD symref's branch/revision is also advertised by `--heads`, otherwise from an advertised head, while leaving `synced_content_file_count` unset and writing no manifest/snapshot/search-index artifacts. Blank, credential-bearing HTTP(S), SSH-URL, or scp-like remotes, unreachable, empty, no-branch, malformed, oversized, combined stdout/stderr-over-cap, malformed-HEAD-symref, or unadvertised-HEAD-symref remotes fail closed with an operator-visible `generic Git repository sync execution failed` error.
- Review-agent execution and repository-sync execution for providers other than configured local/Generic Git remain stub outcomes selected from environment configuration.
- Deferred follow-up work includes real fetch/import execution, broad automated retries/backoff, production scheduling/supervision, durable worker metadata beyond the shared state file and optional supervisor status snapshot, and production-grade observability; the bounded multi-tick loop only repeats the existing tick path under explicit operator control, repository-sync ticks fail malformed queued-job timestamps and recover stale or malformed `running` jobs before claiming more work, and the automatic retry baseline is limited to one replacement for old recovered lease failures (stale or malformed running-lease records).

## Acceptance scenarios
1. Running the real `sourcebot-worker` binary with a resolved organization-state path must log a startup runtime-baseline line that names:
   - the resolved `organization_state_path`
   - the selected review-agent stub outcome
   - the selected repository-sync stub outcome
   - configured bounded tick count and idle sleep
2. When no queued review-agent run or repository-sync job is available, the worker must exit successfully and log the no-work path instead of claiming progress.
3. When a queued review-agent run exists, the default invocation may claim and persist only that single oldest queued run's configured stub terminal outcome before exiting.
4. When no review-agent run is available but a queued repository-sync job exists, the default invocation may claim and persist only that single oldest queued repository-sync job before exiting; provider jobs without a matching configured local or Generic Git connection still use the selected stub terminal outcome.
5. A repository-sync job tied to a configured `local` connection must fail closed before spawning Git when `repo_path` is blank, relative, contains control characters, or includes surrounding whitespace; otherwise it must run a timeout-bounded real local Git working-tree preflight plus bounded HEAD/content-discovery/current-branch/tracked-file probes under the same combined stdout/stderr cap before terminal completion. Missing, invalid, empty, empty-tree, malformed-HEAD-revision, hung, non-UTF-8 tracked path, or combined stdout/stderr-over-cap preflight/execution paths must persist `failed` with a `local repository sync preflight failed` or `local repository sync execution failed` error and must not leave a successful tracked-content manifest/snapshot, while a real local Git repository with a readable current revision, at least one tracked content path, and either a current branch or a confirmed detached `HEAD` label may persist `succeeded` independently of the configured stub failure outcome, record the terminal revision, branch label, and tracked-content file count, write a deterministic repo-local `.sourcebot/local-sync/<organization_id>/<repository_id>/<job_id>/manifest.txt` containing the revision, branch label, and tracked paths from `git ls-tree -rz --name-only HEAD`, materialize a sibling `snapshot/` tree containing only tracked `HEAD` content, preserve valid tab characters in tracked path names, reject manifest-delimiting/control path characters before artifact success, and persist a sibling `search-index.json` artifact that can be loaded without rewalking the snapshot.
6. A repository-sync job tied to a configured `generic_git` connection must fail closed before spawning Git when `base_url` is blank, uses an unsupported URL scheme/transport prefix such as `ext::`, or is an HTTP(S), SSH-URL, or scp-like remote with embedded credentials/userinfo; otherwise it must run timeout-bounded real `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` metadata-only probes with bounded stdout/stderr handling while forcing non-interactive Git authentication and SSH prompt suppression (`GIT_TERMINAL_PROMPT=0`, `GCM_INTERACTIVE=never`, askpass disabled, and SSH `BatchMode=yes`/`NumberOfPasswordPrompts=0`) before terminal completion. A reachable remote with at least one advertised branch must persist terminal revision/current-branch metadata only when the selected branch/revision is advertised by `--heads`, while credential-bearing HTTP(S), unreachable, empty, no-branch, malformed, oversized, combined stdout/stderr-over-cap, malformed or incomplete HEAD-symref, or unadvertised-HEAD-symref remotes must persist `failed` with a `generic Git repository sync execution failed` error and must not claim content-file counts or write local artifacts.
7. The repo-local end-to-end smoke matrix must be able to enqueue a repository-sync job through `POST /api/v1/auth/repository-sync-jobs`, read it as `queued` through `GET /api/v1/auth/repository-sync-jobs`, run the real worker once after higher-priority review-agent work is already drained, and then read the same job as `succeeded` with `started_at` and `finished_at` populated.
8. When `SOURCEBOT_STUB_REPOSITORY_SYNC_JOB_EXECUTION_OUTCOME=failed`, the worker must still exit successfully, persist the failed stub outcome for provider work outside the configured local/Generic Git execution baselines, and log a repository-sync terminal-status line.
9. When `SOURCEBOT_WORKER_MAX_TICKS` is set to a positive integer greater than one, one worker invocation may repeat the same priority-ordered tick path up to that bounded count, process multiple queued jobs across ticks, and log configured bounded runtime completion before exiting; this is still explicit operator-controlled looping, not daemon supervision.
10. Before claiming another repository-sync job, each repository-sync tick must mark queued repository-sync jobs with malformed `queued_at` timestamps as `failed` with an operator-visible timestamp error while preserving valid queued jobs for the same tick's claim path. Each tick must also recover `running` repository-sync jobs whose `started_at` timestamp is at least one hour old by marking them `failed` with an operator-visible lease-expiry error, and must fail closed for malformed `running` lease records with missing or invalid `started_at` timestamps instead of leaving them permanently unclaimable; fresh `running` jobs are preserved before continuing to claim the oldest valid queued job for that tick.
11. A failed stale-lease or malformed-running-lease recovery record whose `finished_at` is at least one hour old may enqueue exactly one automatic replacement job before the tick claims work, but only when no queued/running or prior automatic replacement already exists for the same organization, repository, and connection; this is bounded lease-recovery replay, not general retry/backoff parity.
12. When `SOURCEBOT_WORKER_STATUS_PATH` is set, a successful worker invocation must write a supervisor-readable JSON status snapshot before work, after each bounded tick as a local heartbeat, and after completion. The snapshot contains `schema_version`, `completed`, `updated_at`, `process_id`, `max_ticks`, `ticks_completed`, `last_tick`, `last_outcome`, nullable last work-item id/status/error fields, and nullable repository-sync terminal metadata fields (`last_work_item_synced_revision`, `last_work_item_synced_branch`, and `last_work_item_synced_content_file_count`) so successful local/Generic Git sync ticks expose the persisted revision/branch/content-count summary and failed repository-sync ticks expose a bounded/redacted terminal error summary without log scraping; this is a local operator status artifact, not a supervised daemon or production observability backend.
13. Invalid stub outcome or bounded-loop/status-path configuration must fail closed before execution rather than silently falling back to another outcome.

## Operator-visible runtime contract
- `SOURCEBOT_DATA_DIR` may provide the shared local runtime base directory.
- `SOURCEBOT_ORGANIZATION_STATE_PATH` may explicitly override the organization-state file path.
- `SOURCEBOT_STUB_REVIEW_AGENT_RUN_EXECUTION_OUTCOME` currently supports `completed` and `failed`.
- `SOURCEBOT_STUB_REPOSITORY_SYNC_JOB_EXECUTION_OUTCOME` currently supports `succeeded` and `failed`.
- `SOURCEBOT_WORKER_MAX_TICKS` defaults to `1`; when set, it must be a non-empty positive integer using canonical decimal digits (a leading `+` sign is rejected), with no surrounding whitespace, no greater than `1000000`, and bounds the number of ticks one worker process will run before exiting.
- `SOURCEBOT_WORKER_IDLE_SLEEP_MS` defaults to `1000`; when set, it must be a non-empty non-negative integer using canonical decimal digits (a leading `+` sign is rejected), with no surrounding whitespace, no greater than `3600000`, and controls how long the bounded loop sleeps after a no-work tick before trying the next configured tick.
- `SOURCEBOT_WORKER_STATUS_PATH` is optional; when set to a non-empty path, the worker preflights the path before claiming work, revalidates that existing parents are not symlinks before each snapshot write, opens the final snapshot target with no-follow semantics so a symlink target cannot be followed during the write, refreshes the JSON snapshot after each bounded tick with an RFC3339 `updated_at` heartbeat timestamp and worker `process_id`, and writes it again after successful bounded runtime completion so a local supervisor/operator can inspect the last tick/outcome, the last repository-sync terminal revision/branch/content-count metadata when present, and any last repository-sync terminal error summary without scraping logs; status-file error details are bounded and redact text after stable local, Generic Git, or stale/malformed lease-recovery error prefixes before the snapshot can be surfaced through the authenticated worker-status API.
- Worker logs, optional worker status snapshots, and authenticated sync-history responses are the current runtime-baseline evidence for startup resolution, no-work exits, stub terminal outcomes, bounded multi-tick runtime completion, timeout-bounded local-Git preflight plus combined-output-capped persisted revision/content-count/current-branch/tracked-file probe outcomes, bounded repo-local tracked-content manifests/snapshots/search-index artifacts, timeout-bounded Generic Git remote `ls-remote --heads` revision/current-branch metadata outcomes, non-interactive Git/SSH prompt suppression for worker-spawned Git processes, child-process stdout/stderr collection that retains only the configured per-stream output cap plus one byte and treats either individual stream or the combined stdout+stderr payload exceeding the cap as fail-closed, malformed queued timestamp failure before claim, stale-running repository-sync lease recovery before the next claim, malformed running-lease fail-closed recovery before the next claim, one automatic stale-lease retry after a one-hour backoff, authenticated sync-history `retryable: true` visibility for failed admin-visible jobs that have no active same-target job and are eligible for the existing manual retry API, and fail-closed configuration parsing.
- The local-Git worker baseline only proves that a configured local repository path is an actual Git working tree with a readable current revision, at least one tracked content path discoverable from `HEAD`, and current branch before recording terminal sync-job status plus the terminal revision/branch/content-count metadata and writing the bounded manifest/snapshot/search-index artifacts; it does not fetch, import, reindex, mirror, or reconcile repository contents beyond that repo-local tracked content snapshot/search-index baseline.
- The Generic Git worker baseline only proves that a configured `base_url` is reachable through timeout-bounded `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` metadata-only probes with bounded stdout/stderr handling and non-interactive Git/SSH prompt-suppression environment overrides, is not a credential-bearing HTTP(S), SSH-URL, or scp-like remote, advertises well-formed non-duplicate branch refs, and advertises at least one branch including the selected HEAD-symref branch/revision when that symref is present before recording terminal sync-job revision/current-branch metadata; it still does not fetch, mirror, import, index, or persist repository content for Generic Git remotes.

## Explicit deferrals
The current worker runtime acceptance does **not** claim:
- real review-agent execution
- real repository-sync fetch/import/mirror execution beyond local Git working-tree preflight plus revision/content-discovery/current-branch probes, bounded local-only tracked-content manifest/snapshot/search-index artifact status, Generic Git `ls-remote --heads` revision/current-branch metadata probes, stubbed terminal outcomes, or the bounded enqueue-to-history progress smoke
- automated retries/backoff beyond one bounded automatic lease-recovery retry and the bounded manual retry API
- production polling, daemonization, or supervision beyond the explicit bounded multi-tick loop and optional local status snapshot
- durable worker-specific metadata beyond the shared organization-state file and optional local status snapshot
- queue depth metrics, dashboards, alerts, or production-ready observability
- production deployment guidance beyond the local operator/runtime baseline already documented elsewhere
