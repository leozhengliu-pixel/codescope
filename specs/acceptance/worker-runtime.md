# Worker Runtime Acceptance

## Purpose
This document defines the current clean-room acceptance contract for the `sourcebot-worker` runtime in `crates/worker/src/main.rs`. It describes the shipped one-tick default plus bounded multi-tick, mostly stub-oriented worker baseline with bounded local-Git and Generic Git metadata probes, without over-claiming real fetch/import/reindex execution, retries, production scheduling, durable worker metadata, or production observability.

## Grounding and limits
- Grounded in:
  - `crates/worker/src/main.rs`
  - `crates/worker/src/lib.rs`
  - `crates/worker/tests/worker_runtime_logging_smoke.rs`
  - existing focused worker smoke coverage in `crates/worker/tests/*.rs`
- By default, the worker remains a one-tick local bring-up path. Setting `SOURCEBOT_WORKER_MAX_TICKS=<n>` lets one invocation run a bounded multi-tick loop and then exit; `SOURCEBOT_WORKER_IDLE_SLEEP_MS=<ms>` controls the sleep between no-work ticks.
- The repo-local smoke matrix now proves the API enqueue/history surface can hand a queued repository-sync job to a later `sourcebot-worker` invocation; authenticated local import also queues a newly imported local repository for that same worker baseline after adding the imported repository to the admin organization visibility set. The terminal `succeeded` progress remains observable through the authenticated history API and persisted organization state.
- Repository-sync jobs tied to a configured `local` connection now validate that `repo_path` is non-blank before spawning Git, then run a timeout-bounded real Git preflight (`git -C <repo_path> rev-parse --is-inside-work-tree`) plus bounded revision, content-discovery, and current-branch probes (`git rev-parse HEAD`, `git ls-tree -r --name-only HEAD`, and `git symbolic-ref --short HEAD`) before falling back to stub outcomes: blank, missing, invalid, empty, empty-tree, detached, or hung local repositories fail closed with operator-visible `local repository sync preflight failed` or `local repository sync execution failed` errors instead of claiming success. Successful local jobs write a deterministic repo-local tracked-content manifest plus bounded text snapshot/search-index artifacts under `.sourcebot/local-sync/...` for tracked non-binary files only.
- Repository-sync jobs tied to a configured `generic_git` connection now validate that `base_url` is non-blank and that HTTP(S) URLs do not include embedded credentials before spawning Git, then run timeout-bounded real `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` metadata probes before falling back to stub outcomes. Reachable remotes with at least one advertised branch persist terminal revision/current-branch metadata from the default branch when advertised, otherwise from an advertised head, while leaving `synced_content_file_count` unset and writing no manifest/snapshot/search-index artifacts. Blank, credential-bearing HTTP(S), unreachable, empty, no-branch, malformed, oversized, or malformed-HEAD-symref remotes fail closed with an operator-visible `generic Git repository sync execution failed` error.
- Review-agent execution and repository-sync execution for providers other than configured local/Generic Git remain stub outcomes selected from environment configuration.
- Deferred follow-up work includes real fetch/import execution, broad automated retries/backoff, production scheduling/supervision, durable worker metadata beyond the shared state file and optional supervisor status snapshot, and production-grade observability; the bounded multi-tick loop only repeats the existing tick path under explicit operator control, repository-sync ticks recover stale `running` jobs after a bounded lease before claiming more work, and the automatic retry baseline is limited to one queued retry for stale-lease failures after an additional one-hour backoff when no queued/running or prior automatic replacement exists.

## Acceptance scenarios
1. Running the real `sourcebot-worker` binary with a resolved organization-state path must log a startup runtime-baseline line that names:
   - the resolved `organization_state_path`
   - the selected review-agent stub outcome
   - the selected repository-sync stub outcome
   - configured bounded tick count and idle sleep
2. When no queued review-agent run or repository-sync job is available, the worker must exit successfully and log the no-work path instead of claiming progress.
3. When a queued review-agent run exists, the default invocation may claim and persist only that single oldest queued run's configured stub terminal outcome before exiting.
4. When no review-agent run is available but a queued repository-sync job exists, the default invocation may claim and persist only that single oldest queued repository-sync job before exiting; provider jobs without a matching configured local or Generic Git connection still use the selected stub terminal outcome.
5. A repository-sync job tied to a configured `local` connection must fail closed before spawning Git when `repo_path` is blank; otherwise it must run a timeout-bounded real local Git working-tree preflight plus bounded HEAD/content-discovery/current-branch probes before terminal completion. Missing, invalid, empty, empty-tree, detached, or hung paths must persist `failed` with a `local repository sync preflight failed` or `local repository sync execution failed` error and must not leave a successful tracked-content manifest/snapshot, while a real local Git repository with a readable current revision, at least one tracked content path, and current branch may persist `succeeded` independently of the configured stub failure outcome, record the terminal revision, branch, and tracked-content file count, write a deterministic repo-local `.sourcebot/local-sync/<organization_id>/<repository_id>/<job_id>/manifest.txt` containing the revision, branch, and tracked paths from `git ls-tree -r --name-only HEAD`, materialize a sibling `snapshot/` tree containing only tracked `HEAD` content, and persist a sibling `search-index.json` artifact that can be loaded without rewalking the snapshot.
6. A repository-sync job tied to a configured `generic_git` connection must fail closed before spawning Git when `base_url` is blank or is an HTTP(S) URL with embedded credentials; otherwise it must run timeout-bounded real `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` probes before terminal completion. A reachable remote with at least one advertised branch must persist terminal revision/current-branch metadata only, while credential-bearing HTTP(S), unreachable, empty, no-branch, malformed, oversized, or malformed-HEAD-symref remotes must persist `failed` with a `generic Git repository sync execution failed` error and must not claim content-file counts or write local artifacts.
7. The repo-local end-to-end smoke matrix must be able to enqueue a repository-sync job through `POST /api/v1/auth/repository-sync-jobs`, read it as `queued` through `GET /api/v1/auth/repository-sync-jobs`, run the real worker once after higher-priority review-agent work is already drained, and then read the same job as `succeeded` with `started_at` and `finished_at` populated.
8. When `SOURCEBOT_STUB_REPOSITORY_SYNC_JOB_EXECUTION_OUTCOME=failed`, the worker must still exit successfully, persist the failed stub outcome for provider work outside the configured local/Generic Git execution baselines, and log a repository-sync terminal-status line.
9. When `SOURCEBOT_WORKER_MAX_TICKS` is set to a positive integer greater than one, one worker invocation may repeat the same priority-ordered tick path up to that bounded count, process multiple queued jobs across ticks, and log configured bounded runtime completion before exiting; this is still explicit operator-controlled looping, not daemon supervision.
10. Before claiming another repository-sync job, each repository-sync tick must recover `running` repository-sync jobs whose `started_at` timestamp is at least one hour old by marking them `failed` with an operator-visible lease-expiry error, and must fail closed for malformed `running` lease records with missing or invalid `started_at` timestamps instead of leaving them permanently unclaimable; fresh `running` jobs are preserved before continuing to claim the oldest queued job for that tick.
11. A failed stale-lease recovery record whose `finished_at` is at least one hour old may enqueue exactly one automatic replacement job before the tick claims work, but only when no queued/running or prior automatic replacement already exists for the same organization, repository, and connection; this is bounded stale-lease replay, not general retry/backoff parity.
12. When `SOURCEBOT_WORKER_STATUS_PATH` is set, a successful worker invocation must write a supervisor-readable JSON status snapshot before work, after each bounded tick as a local heartbeat, and after completion. The snapshot contains `schema_version`, `completed`, `updated_at`, `process_id`, `max_ticks`, `ticks_completed`, `last_tick`, `last_outcome`, and nullable last work-item id/status/error fields so failed repository-sync ticks expose a bounded/redacted terminal error summary without log scraping; this is a local operator status artifact, not a supervised daemon or production observability backend.
13. Invalid stub outcome or bounded-loop/status-path configuration must fail closed before execution rather than silently falling back to another outcome.

## Operator-visible runtime contract
- `SOURCEBOT_DATA_DIR` may provide the shared local runtime base directory.
- `SOURCEBOT_ORGANIZATION_STATE_PATH` may explicitly override the organization-state file path.
- `SOURCEBOT_STUB_REVIEW_AGENT_RUN_EXECUTION_OUTCOME` currently supports `completed` and `failed`.
- `SOURCEBOT_STUB_REPOSITORY_SYNC_JOB_EXECUTION_OUTCOME` currently supports `succeeded` and `failed`.
- `SOURCEBOT_WORKER_MAX_TICKS` defaults to `1`; when set, it must be a positive integer and bounds the number of ticks one worker process will run before exiting.
- `SOURCEBOT_WORKER_IDLE_SLEEP_MS` defaults to `1000`; when set, it must be a non-negative integer and controls how long the bounded loop sleeps after a no-work tick before trying the next configured tick.
- `SOURCEBOT_WORKER_STATUS_PATH` is optional; when set to a non-empty path, the worker preflights the path before claiming work, refreshes the JSON snapshot after each bounded tick with an RFC3339 `updated_at` heartbeat timestamp and worker `process_id`, and writes it again after successful bounded runtime completion so a local supervisor/operator can inspect the last tick/outcome and any last repository-sync terminal error summary without scraping logs; status-file error details are bounded and redact text after the stable error prefix before the snapshot can be surfaced through the authenticated worker-status API.
- Worker logs, optional worker status snapshots, and authenticated sync-history responses are the current runtime-baseline evidence for startup resolution, no-work exits, stub terminal outcomes, bounded multi-tick runtime completion, timeout-bounded local-Git preflight plus persisted revision/content-count/current-branch probe outcomes, bounded repo-local tracked-content manifests/snapshots/search-index artifacts, timeout-bounded Generic Git remote `ls-remote --heads` revision/current-branch metadata outcomes, child-process stdout/stderr collection that retains only the configured output cap plus one byte and terminates the child promptly once either stream exceeds the cap, stale-running repository-sync lease recovery before the next claim, malformed running-lease fail-closed recovery before the next claim, one automatic stale-lease retry after a one-hour backoff, authenticated sync-history `retryable: true` visibility for failed admin-visible jobs that have no active same-target job and are eligible for the existing manual retry API, and fail-closed configuration parsing.
- The local-Git worker baseline only proves that a configured local repository path is an actual Git working tree with a readable current revision, at least one tracked content path discoverable from `HEAD`, and current branch before recording terminal sync-job status plus the terminal revision/branch/content-count metadata and writing the bounded manifest/snapshot/search-index artifacts; it does not fetch, import, reindex, mirror, or reconcile repository contents beyond that repo-local tracked content snapshot/search-index baseline.
- The Generic Git worker baseline only proves that a configured `base_url` is reachable through timeout-bounded `git ls-remote --symref <base_url> HEAD` and `git ls-remote --heads <base_url>` probes, is not a credential-bearing HTTP(S) URL, and advertises at least one branch before recording terminal sync-job revision/current-branch metadata; it does not clone, fetch, import, reindex, mirror, discover repositories, count content files, or write manifest/snapshot/search-index artifacts.

## Explicit deferrals
The current worker runtime acceptance does **not** claim:
- real review-agent execution
- real repository-sync fetch/import/mirror execution beyond local Git working-tree preflight plus revision/content-discovery/current-branch probes, bounded local-only tracked-content manifest/snapshot/search-index artifact status, Generic Git `ls-remote --heads` revision/current-branch metadata probes, stubbed terminal outcomes, or the bounded enqueue-to-history progress smoke
- automated retries/backoff beyond one bounded automatic stale-lease retry and the bounded manual retry API
- production polling, daemonization, or supervision beyond the explicit bounded multi-tick loop and optional local status snapshot
- durable worker-specific metadata beyond the shared organization-state file and optional local status snapshot
- queue depth metrics, dashboards, alerts, or production-ready observability
- production deployment guidance beyond the local operator/runtime baseline already documented elsewhere
