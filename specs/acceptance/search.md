# Acceptance Spec: Search

## Scope
- Dedicated `#/search` frontend route for API-backed code search
- Cross-repo and cross-branch code search
- Literal, regex, and boolean-style query modes
- Repo/language/path filters
- Snippet results with file and revision context

## Inputs
- User query string
- Optional search mode: literal | regex | boolean
- Optional filters: repositories, branches, languages, path globs
- Authenticated user context

## Expected behavior
1. The top-level frontend shell exposes a dedicated `#/search` route so code search is reachable without returning to the repository home inventory.
2. That route reuses the existing API-backed search flow with query input, optional repository filter, explicit boolean/literal/regex mode selection, and result rendering grounded in `/api/v1/search`.
3. A valid query returns ranked matches across all repositories the user can access; the current local/index-artifact baseline accepts explicit `mode=boolean|literal|regex`, fails closed for empty/whitespace-only regex queries instead of matching every indexed line, matches boolean query text case-insensitively, requires every unquoted space-separated query term within a term group to appear on a result line, treats an unquoted `OR` token as a bounded boolean separator between line-term groups, fails closed for oversized boolean queries before allocating unbounded OR groups, fails closed for oversized literal-mode queries before allocating or lowercasing unbounded query strings, keeps quoted phrases together as literal line substrings, applies bounded unquoted `lang:` filters (including `lang:typescript`/`lang:ts` for `.ts` files and common config/source extensions such as Java, JSON, YAML/YML, and TOML) plus substring and glob-style `path:` filters (`*`, `?`, and recursive `**`) to result file paths, and returns deterministic repo/path/line ordering for identical indexed content (including file-vs-directory name prefix collisions such as `a.rs` before `a/file.rs`) and for primary-index results or, when primary search is empty/unavailable, caller-authorized local-sync artifact fallback results before pagination, while preserving the caller's trimmed query text and selected mode in the response.
4. Each result includes repository, revision or branch context, file path, line-oriented snippet, and enough metadata to open the file view.
5. Filters narrow the result set without leaking inaccessible repositories.
6. Authenticated users can create, list, select, and delete saved search contexts through `/api/v1/auth/search-contexts`; the dedicated `#/search` route exposes a bounded management baseline for caller-owned contexts with non-empty repository scopes, and `GET /api/v1/search?context_id=...` applies the saved context as an additional repository-scope filter.
7. Empty-result queries return a successful response with zero hits, not an internal error.
8. Invalid regex queries return a user-safe validation error.
9. `/api/v1/search` accepts bounded `limit` and `offset` parameters, rejects `limit=0`, `limit>500`, and `offset>10000` with `400 Bad Request`, and returns a `pagination` object containing `limit`, `offset`, visibility-scoped `total_count`, and `has_more` for the current query; the shared search store also exposes a bounded page helper with the same pagination metadata so runtime callers do not need to hand-roll result slicing.
10. `/api/v1/search` rejects empty queries and queries over 1024 bytes with `400 Bad Request` after authentication so oversized search input cannot force unbounded runtime matching work.
11. Pagination is stable for the same repository revision and query parameters.

## Permission behavior
- Results from repositories outside the caller's permissions must not appear in results, pagination counts, snippets, suggestions, or facets.
- Search contexts must only expand to repositories visible to the caller.
- Saved search contexts are caller-owned. Unknown, deleted, other-user, or present-but-blank context ids fail closed instead of broadening search, and an explicit `repo_id` outside the saved context scope fails closed.
- Explicit `repo_id` filters are fail-closed: hidden, unknown, or present-but-blank repository ids return `404` instead of being treated as absent and broadening the search across all visible repositories; when a requested-repo backend or local-sync fallback returns mixed visible-repository rows, the API retains only rows whose `repo_id` equals the requested repository before computing the response, and local-sync fallback remains eligible when that requested repository is visible through more than one caller-authorized organization.
- Persisted local-sync search index artifacts must fail closed when the artifact path is not a direct regular file (including symlinks), when their required schema version is missing or unsupported, when their embedded `index_status.repo_id` differs from the repository id requested by the loader, when `index_status.indexed_line_count` disagrees with the stored indexed-line payload, when `index_status.indexed_file_count` undercounts the unique stored result paths, when `indexed` status reports zero indexed files, when `indexed_empty` status reports non-zero indexed files or lines, when `error` status still carries indexed files/lines/content or lacks a non-empty operator-visible error message, when `indexed`/`indexed_empty` success status carries a stale error message, when any stored result path is oversized, absolute, escapes via `..`, contains blank path components, line separators, or non-tab control characters, uses a non-normal separator form, or targets a runtime/generated/binary path that the live indexer would skip (for example `.sourcebot`, `target`, `node_modules`, `dist`, or skipped extensions), when any stored result has binary/NUL line content, when any stored result has line number zero, when stored path/line keys are duplicate or not sorted by path then line number, or when stored line numbers for one path are non-contiguous; corrupted artifacts are not trusted for search result paths or repository status identity/counts. Artifact-backed search derives local-sync artifact paths with non-colliding encoded organization/repository/job path components so unsafe ids such as `job/latest` cannot resolve to stale sibling artifacts like `job_latest`; it fails closed for present malformed or non-regular artifacts instead of following indirection, falling back to mutable snapshot contents, returning an empty success for the affected repo, or silently skipping the bad artifact into partial unscoped/search-context success for other visible repositories, while index-status surfaces malformed or non-regular artifacts as an `error` status with a redacted `malformed local-sync search index artifact` parse/load reason and surfaces valid artifact `error` statuses without treating them as searchable content. Search treats a primary startup index with `error` status as unavailable for fallback purposes, so a visible requested repository can still use its latest authorized local-sync artifact instead of returning an empty success from the failed primary index. Cross-repository search may fall back to caller-authorized local-sync artifact results for visible repositories missing from the primary startup index only when the primary startup search response is empty; it does not merge local-sync artifacts into a non-empty primary response.
- This saved-context, pagination, and local search-matching baseline now includes API metadata plus bounded `#/search` previous/next controls for result pages, frontend boolean/literal/regex mode selection with deep-link preservation, authenticated listing, creation from a specific non-empty repository filter, selection, and deletion of caller-owned saved search contexts on `#/search` that sends non-blank `context_id` values to `/api/v1/search`, explicit blank-`repo_id` fail-closed behavior, authenticated `400 Bad Request` handling for empty or over-1024-byte search queries, explicit `mode=boolean|literal|regex` matching, fail-closed local regex matching for empty/whitespace-only patterns, over-1024-byte patterns, and regex programs over the configured compiled-size limit, fail-closed local boolean matching for oversized OR-heavy queries, fail-closed local literal matching for oversized query strings, repository-root-relative skipping of `.sourcebot` runtime metadata so local-sync artifacts are not indexed as source files, case-insensitive boolean text matching, all-term matching for unquoted space-separated query terms within boolean groups, quoted-phrase components, a bounded unquoted `OR` separator between line-term groups, minimal unquoted `lang:` filters for Rust, TypeScript/JavaScript, Python, Markdown, Go, JSON, YAML/YML, and TOML plus substring and glob-style `path:` result-path filters whose oversized `path:` glob payloads fail closed before matcher allocation, fail-closed persisted-artifact non-regular/symlink path, required-schema-version validation, non-colliding encoded local-sync artifact path components, symlinked parent write rejection before nested directory creation, symlinked temp-artifact write rejection without clobbering external targets, writer-side status/content consistency validation before replacing the last valid artifact, status repo-id, indexed-line-count, indexed-file-under-count, indexed-without-files state, indexed-empty state, error-status-with-content validation, error-status-without-operator-message validation, success-status-with-stale-error validation, binary-line-content validation, changing indexes/revisions.

## Edge cases
- Very large repositories must remain searchable without loading full trees into memory.
- Binary/generated extensions are excluded by default regardless of extension casing.
- Queries that hit generated/minified/vendor content must be suppressible by policy.
- Branches missing an index should surface partial availability rather than silent omission.

## Black-box examples
- Opening `#/search` shows a dedicated code-search page with the existing query input, repository filter, boolean/literal/regex mode selector, bounded result-page summary, previous/next controls backed by `/api/v1/search?limit=20&offset=...`, and direct links into `#/repos/:repoId?path=...&from=search` so a user can jump from a match into repository source without losing obvious search-route context.
- Query `router` across all accessible repos returns matches from multiple repos.
- Query `lang:rust path:crates/api healthz` returns matches only under `crates/api` in Rust files.
- Query with invalid regex `([a-z` returns a validation error with no server crash.
